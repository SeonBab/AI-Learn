# 커밋 메시지 컨벤션

## 설명

이 문서는 프로젝트에서 사용할 커밋 제목과 커밋 본문 작성 규칙을 정리한다.

기본 형식은 Conventional Commits 스타일을 참고하되, C++ 학습용 프로젝트에서 읽기 쉽고 관리하기 쉬운 형태로 단순화한다.

## 핵심 정리

### 1. 기본 형식

```text
type: 짧은 변경 요약

변경한 이유와 주요 내용을 설명한다.
필요하면 여러 줄로 작성한다.
```

작은 변경은 제목만으로 충분하면 본문을 생략할 수 있다.

```text
docs: 커밋 메시지 컨벤션 문서 추가
```

### 2. 커밋 제목 규칙

커밋 제목은 다음 형식을 사용한다.

```text
type: 변경 내용 요약
```

제목 작성 규칙:

1. 제목은 한 줄로 작성한다.
2. 가능하면 50자 안팎으로 짧게 작성한다.
3. 마침표를 붙이지 않는다.
4. 무엇을 변경했는지 명확하게 작성한다.
5. 한글을 기본으로 사용한다.
6. 타입은 영어 소문자로 작성한다.

좋은 예:

```text
docs: 프로젝트 코딩 컨벤션 문서 추가
feat: 채팅 메시지 저장 기능 추가
fix: 빈 메시지 전송 오류 수정
refactor: 메시지 파싱 로직 분리
test: 채팅 세션 테스트 추가
```

### 3. 커밋 타입

| 타입 | 사용 시점 | 예시 |
|---|---|---|
| `feat` | 새로운 기능 추가 | `feat: 채팅 메시지 저장 기능 추가` |
| `fix` | 버그 수정 | `fix: 빈 메시지 전송 오류 수정` |
| `docs` | 문서 추가 또는 수정 | `docs: 커밋 컨벤션 문서 추가` |
| `style` | 코드 의미 변화 없는 형식 수정 | `style: 들여쓰기 정리` |
| `refactor` | 동작 변화 없는 구조 개선 | `refactor: 메시지 처리 로직 분리` |
| `test` | 테스트 추가 또는 수정 | `test: 채팅 세션 테스트 추가` |
| `chore` | 빌드, 설정, 기타 작업 | `chore: gitignore 설정 추가` |
| `build` | 빌드 시스템 또는 의존성 변경 | `build: CMake 설정 추가` |
| `ci` | CI 설정 변경 | `ci: 테스트 워크플로 추가` |
| `perf` | 성능 개선 | `perf: 메시지 검색 속도 개선` |
| `revert` | 이전 커밋 되돌리기 | `revert: 메시지 저장 기능 되돌리기` |

### 4. 커밋 본문 규칙

본문은 커밋 제목만으로 설명이 부족할 때 작성한다.

본문에는 변경 이유, 변경 내용, 영향 범위, 주의 사항을 포함할 수 있다.

```text
feat: 채팅 메시지 저장 기능 추가

사용자와 assistant의 대화 내용을 세션 단위로 보관할 수 있도록
FChatMessage와 FChatSession 구조를 추가했다.

이 변경으로 이후 대화 기록 조회와 문서 저장 기능을 구현할 수 있다.
```

### 5. scope 사용 규칙

변경 범위를 더 명확히 하고 싶을 때는 scope를 사용할 수 있다.

```text
type(scope): 변경 내용 요약
```

예시:

```text
docs(convention): 커밋 메시지 규칙 추가
feat(chat): 메시지 저장 기능 추가
fix(parser): 빈 입력 처리 오류 수정
```

권장 scope는 `docs`, `convention`, `chat`, `cpp`, `test`, `build`이다.

### 6. Breaking Change 규칙

기존 사용 방식을 깨는 변경은 제목이나 본문에 명확히 표시한다.

```text
feat!: 채팅 세션 저장 형식 변경
```

또는 본문에 다음처럼 적는다.

```text
BREAKING CHANGE: 기존 MessageList 기반 저장 형식을 SessionHistory 기반 저장 형식으로 변경했다.
```

### 7. 커밋 단위 규칙

커밋은 하나의 논리적 변경 단위로 작성한다.

1. 문서 변경과 코드 변경은 가능하면 분리한다.
2. 기능 추가와 버그 수정은 분리한다.
3. 리팩터링과 동작 변경은 분리한다.
4. 관련 없는 파일 변경을 같은 커밋에 넣지 않는다.

### 8. 커밋 전 확인 사항

1. 커밋 제목이 `type: 요약` 형식인지 확인한다.
2. 제목이 너무 모호하지 않은지 확인한다.
3. 한 커밋에 관련 없는 변경이 섞이지 않았는지 확인한다.
4. 문서 파일은 기본적으로 한글로 작성되었는지 확인한다.
5. C++ 예제 코드는 `cpp` 코드 블록을 사용했는지 확인한다.
6. 빌드 또는 테스트가 필요한 변경이면 가능한 범위에서 검증했는지 확인한다.
