# C++ 로컬 AI 기반 텍스트 MUD 게임 기획

## 설명

현재 프로젝트 목표는 1단계 로컬 LLM 채팅 프로그램을 기반으로, C++ 학습용 텍스트 MUD 게임을 확장 제작하는 것이다.

기존 로컬 AI 채팅 프로그램은 게임 안에서 AI NPC 응답을 생성하는 기능으로 재사용한다. 최종 목표는 거대한 온라인 MUD가 아니라, C++ 기초와 객체 설계, 파일 입출력, 로컬 LLM 연동 구조를 함께 학습할 수 있는 작은 싱글플레이 텍스트 RPG를 완성하는 것이다.

## 핵심 정리

1. 첫 번째 목표는 콘솔에서 명령어를 입력하고 결과를 확인할 수 있는 작은 플레이 루프를 완성하는 것이다.
2. 게임의 기본 흐름은 `명령어 입력 -> 게임 상태 변경 -> 결과 출력 -> AI NPC 대화`로 잡는다.
3. 로컬 LLM은 직접 게임 상태를 변경하지 않고, NPC 대사와 힌트 생성을 담당한다.
4. 게임 상태 변경은 반드시 C++ 게임 로직에서 처리한다.
5. 초기 버전은 탐험, 이동, 아이템, NPC 대화, 저장과 불러오기에 집중한다.

## 프로젝트 학습 목표

이 프로젝트에서 학습할 핵심 주제는 다음과 같다.

| 학습 주제 | 설명 |
|---|---|
| C++ 클래스와 객체 설계 | `Room`, `Item`, `NPC`, `Player`, `World`, `GameEngine` 같은 객체를 설계한다. |
| 문자열 명령어 처리 | `look`, `go north`, `take old_key` 같은 명령어를 파싱한다. |
| 자료구조 사용 | 방 목록, 아이템 목록, 인벤토리, NPC 목록을 관리한다. |
| 게임 상태 관리 | 현재 위치, 보유 아이템, 잠금 해제 상태, 퀘스트 진행도를 관리한다. |
| 파일 저장과 불러오기 | 현재 게임 상태를 파일로 저장하고 다시 불러온다. |
| 로컬 LLM API 호출 | Ollama 또는 로컬 LLM에 NPC 대화 요청을 보낸다. |
| 확장 가능한 구조 설계 | 나중에 네트워크 게임으로 확장할 수 있도록 입력, 로직, 출력 계층을 분리한다. |

## 초기 플레이어 명령어

초기 버전에서 지원할 명령어는 다음과 같다.

| 행동 | 설명 | 예시 명령어 |
|---|---|---|
| 현재 위치 확인 | 현재 방의 설명을 본다. | `look` |
| 이동 | 다른 방으로 이동한다. | `go north` |
| 아이템 확인 | 방에 있는 아이템을 본다. | `items` |
| 아이템 줍기 | 방의 아이템을 인벤토리에 넣는다. | `take old_key` |
| 인벤토리 확인 | 가진 아이템 목록을 본다. | `inventory` |
| 아이템 사용 | 특정 아이템을 사용한다. | `use old_key` |
| NPC 확인 | 현재 방의 NPC 목록을 본다. | `npcs` |
| NPC와 대화 | 고정 대사 NPC에게 말을 건다. | `talk guard` |
| AI NPC에게 질문 | 로컬 LLM으로 자연어 응답을 받는다. | `ask guard 성 안에는 뭐가 있나요?` |
| 도움말 보기 | 사용 가능한 명령어를 본다. | `help` |
| 게임 저장 | 현재 상태를 파일에 저장한다. | `save` |
| 게임 불러오기 | 저장된 상태를 불러온다. | `load` |
| 게임 종료 | 프로그램을 종료한다. | `quit` |

초기 구현에서는 전투, 체력, 레벨, 경험치, 장비, 상점 구매, 복잡한 퀘스트, 멀티플레이, 실시간 채팅, 서버 접속은 제외한다.

## 기본 월드 구성

MUD 게임의 기본 단위는 `Room`이다.

각 방은 다음 정보를 가진다.

| 요소 | 설명 |
|---|---|
| 방 ID | 내부에서 방을 구분하는 고유 이름 |
| 방 이름 | 플레이어에게 보여줄 장소 이름 |
| 방 설명 | `look` 명령어 입력 시 출력되는 설명 |
| 연결된 방 | 북쪽, 남쪽, 동쪽, 서쪽 등 이동 가능한 방향 |
| 아이템 목록 | 방 안에 놓여 있는 아이템 |
| NPC 목록 | 방 안에 존재하는 NPC |
| 잠김 여부 | 특정 조건을 만족해야 들어갈 수 있는지 여부 |

초기 맵은 3개에서 5개 방으로 제한한다.

```text
잡화점 <- 마을 광장 -> 우물가
              |
          성문 앞
              |
          성 안뜰
```

초기 시나리오 제목은 임시로 `낡은 성문의 비밀`로 정한다.

플레이어는 작은 마을에서 시작한다. 마을 북쪽 또는 남쪽에는 잠긴 성문이 있고, 성문을 열려면 낡은 열쇠가 필요하다. 열쇠는 잡화점에 있으며, 경비병과 상인은 플레이어에게 힌트를 제공한다. 플레이어가 열쇠를 얻고 성문을 열면 첫 번째 목표를 달성한다.

## 아이템과 인벤토리

아이템은 플레이어가 줍거나 사용할 수 있는 오브젝트다.

초기 아이템 정보는 다음 정도로 제한한다.

| 요소 | 설명 |
|---|---|
| 아이템 ID | 내부 식별자 |
| 이름 | 출력용 이름 |
| 설명 | 아이템 설명 |
| 사용 가능 여부 | 사용할 수 있는 아이템인지 여부 |
| 효과 | 사용했을 때 발생하는 결과 |

초기 아이템 예시는 다음과 같다.

```text
ID: old_key
이름: 낡은 열쇠
설명: 성문을 열 수 있을 것 같은 오래된 열쇠다.
효과: 성문 잠금 해제
```

초기 인벤토리는 `vector<Item>` 또는 `vector<string>`으로 구현할 수 있다. 아이템 수량 시스템은 나중에 확장한다.

## NPC와 로컬 LLM 대화

NPC는 두 종류로 나눈다.

| NPC 종류 | 역할 |
|---|---|
| 고정 대사 NPC | 미리 작성한 대사를 출력하고 퀘스트 힌트를 제공한다. |
| AI NPC | 로컬 LLM을 호출해 현재 게임 상황에 맞는 자연어 응답을 생성한다. |

AI NPC는 힌트 제공, 세계관 설명, 자연스러운 대화, 퀘스트 분위기 강화를 담당한다.

AI NPC 응답 규칙은 다음과 같다.

```text
1. NPC의 입장에서만 대답한다.
2. 게임에 없는 장소, 아이템, NPC를 새로 만들지 않는다.
3. 플레이어에게 직접 명령어를 실행해주지 않는다.
4. 게임 상태를 변경했다고 말하지 않는다.
5. 답변은 3문장 이내로 한다.
6. 힌트는 줄 수 있지만 정답을 너무 직접적으로 말하지 않는다.
```

로컬 LLM 호출 흐름은 다음과 같다.

```text
사용자 입력
-> C++ 명령어 파서
-> ask 명령어인지 확인
-> NPC 정보와 게임 상태를 프롬프트로 구성
-> Ollama 또는 로컬 LLM에 요청
-> 응답을 NPC 대사로 출력
```

## 저장과 불러오기

초기 버전부터 저장 기능을 포함한다.

저장 대상은 다음 정도로 제한한다.

| 저장 항목 | 예시 |
|---|---|
| 현재 방 ID | `village_square` |
| 플레이어 인벤토리 | `old_key` |
| 열린 문 상태 | `castle_gate_unlocked=true` |
| 대화 진행 상태 | `guard_talked=true` |
| 퀘스트 상태 | `main_quest_stage=1` |

C++ 기초 학습 단계에서는 단순 텍스트 파일로 시작하고, 구조가 복잡해지면 JSON 라이브러리를 도입한다.

```text
current_room=village_square
inventory=old_key
castle_gate_unlocked=true
guard_talked=true
main_quest_stage=1
```

## 권장 C++ 구조

나중에 네트워크 게임으로 확장할 수 있도록 입력 처리, 게임 로직, 출력 처리를 분리한다.

```text
입력
-> CommandParser
-> GameEngine
-> 결과 메시지 반환
-> 콘솔 출력
```

권장 클래스는 다음과 같다.

| 클래스 | 역할 |
|---|---|
| `CommandParser` | 사용자 입력 문자열을 명령어로 해석한다. |
| `GameEngine` | 명령어를 처리하고 게임 상태를 변경한다. |
| `World` | 방, 아이템, NPC 같은 고정 월드 데이터를 가진다. |
| `Room` | 장소 정보와 연결 관계를 가진다. |
| `Item` | 아이템 ID, 이름, 설명, 사용 효과를 가진다. |
| `NPC` | NPC 이름, 성격, 고정 대사, AI 응답 설정을 가진다. |
| `Player` | 현재 방, 인벤토리, 퀘스트 상태를 가진다. |
| `GameState` | 전체 진행 상태와 저장 대상 데이터를 가진다. |
| `LLMClient` | Ollama 또는 로컬 LLM 호출을 담당한다. |

초기 콘솔 버전에서도 `GameEngine::processCommand(command)`가 문자열 결과를 반환하도록 설계한다. 콘솔 버전은 반환된 문자열을 `cout`으로 출력하고, 네트워크 버전은 같은 문자열을 클라이언트에게 전송할 수 있다.

## 첫 번째 최소 구현 범위

첫 번째 MUD 버전은 다음 기능만 구현한다.

```text
1. 콘솔에서 명령어 입력
2. 3개 이상의 방 생성
3. 방 이동
4. 현재 방 설명 출력
5. 아이템 줍기
6. 인벤토리 확인
7. 아이템 조건으로 잠긴 장소 열기
8. 고정 대사 NPC와 대화
9. 로컬 LLM 기반 AI NPC 응답
10. 게임 저장 및 불러오기
```

가장 먼저 완성해야 할 플레이 루프는 다음과 같다.

```text
look
go east
take old_key
inventory
go west
go south
use old_key
go south
talk guard
ask guard 성 안에는 뭐가 있나요?
save
quit
```
